What Is MCP, Actually?

March 10, 2026 · 10 min read

I keep seeing "MCP" everywhere — blog posts, GitHub repos, Twitter threads. Half the explanations are marketing fluff. The other half assume you already know what JSON-RPC is. So here's my attempt at the version I wish I'd read six months ago: what MCP actually is, what problem it solves, and what the messages look like on the wire. With real examples you can run yourself.

The one-sentence version

MCP (Model Context Protocol) is a way for AI assistants to use tools. Your IDE talks to an MCP server over stdin/stdout, and that server can do things — fetch URLs, read files, query databases, whatever. The AI decides when to call a tool. The MCP server decides how to execute it.

Why does this need a protocol?

Before MCP, every AI tool integration was bespoke. Want Claude to search the web? Custom plugin. Want it to read your filesystem? Different custom plugin. Want it to talk to your company's API? Build something from scratch.

The problem isn't building one integration. It's building twenty. And then maintaining them across IDE updates, API changes, and new AI models. It doesn't scale.

MCP standardizes this. One protocol, one config format, and suddenly the same server works with Claude Desktop, Cursor, VS Code Copilot, and Windsurf. Write it once, use it everywhere.

What it looks like in practice

You have a JSON config file that tells your IDE which MCP servers to run. Here's a real one:

{
  "mcpServers": {
    "fetch": {
      "command": "uvx",
      "args": ["mcp-server-fetch"]
    },
    "github": {
      "command": "uvx",
      "args": ["mcp-server-github"],
      "env": {
        "GITHUB_TOKEN": "ghp_..."
      }
    },
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/home/user/projects"]
    }
  }
}

Three servers. fetch can grab web pages. github can interact with repos, issues, PRs. filesystem can read and write files in a specific directory. Each one is a separate process that your IDE launches and talks to.

When you open Claude Desktop (or Cursor, or whatever), it reads this config, starts each server as a subprocess, and runs through a handshake. After that, the AI can call any of these tools when it decides it needs them.

The handshake — what actually happens when a server starts

This is the part most explanations skip. Let me show you the actual messages. I'll use mcptools to inspect a real server, but I'll show you the raw JSON-RPC underneath.

Step 1: Your IDE starts the server process and sends an initialize message over stdin:

{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "initialize",
  "params": {
    "protocolVersion": "2024-11-05",
    "capabilities": {},
    "clientInfo": {
      "name": "claude-desktop",
      "version": "1.0.0"
    }
  }
}

This is a JSON-RPC 2.0 request. The id: 1 means the IDE expects a response with the same ID. The protocol version tells the server which version of MCP we're speaking.

Step 2: The server responds with what it can do:

{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "protocolVersion": "2024-11-05",
    "capabilities": {
      "tools": {},
      "prompts": {}
    },
    "serverInfo": {
      "name": "mcp-fetch",
      "version": "1.26.0"
    }
  }
}

The key thing here is capabilities. This server says "I have tools and prompts." It doesn't have resources. So the IDE knows not to ask for resources — it'll only query what the server actually supports.

Step 3: The IDE sends a notification saying "OK, I'm ready":

{
  "jsonrpc": "2.0",
  "method": "notifications/initialized"
}

No id on this one — it's a notification. Fire and forget. No response expected.

Step 4: Now the IDE asks what tools are available:

{
  "jsonrpc": "2.0",
  "id": 2,
  "method": "tools/list"
}

{
  "jsonrpc": "2.0",
  "id": 2,
  "result": {
    "tools": [
      {
        "name": "fetch",
        "description": "Fetches a URL from the internet and extracts its contents as markdown.",
        "inputSchema": {
          "type": "object",
          "properties": {
            "url": { "type": "string" },
            "max_length": { "type": "integer" },
            "raw": { "type": "boolean" }
          },
          "required": ["url"]
        }
      }
    ]
  }
}

That inputSchema is a standard JSON Schema. The AI model reads this and knows: "I can call fetch with a required url string, and optionally max_length and raw." This is how the AI knows what arguments to pass — it's not guessing, it's reading a schema.

When the AI actually calls a tool

Later, during a conversation, the AI decides it needs to fetch a webpage. The IDE sends:

{
  "jsonrpc": "2.0",
  "id": 3,
  "method": "tools/call",
  "params": {
    "name": "fetch",
    "arguments": {
      "url": "https://example.com"
    }
  }
}

And the server responds with the content:

{
  "jsonrpc": "2.0",
  "id": 3,
  "result": {
    "content": [
      {
        "type": "text",
        "text": "# Example Domain\n\nThis domain is for use in illustrative examples..."
      }
    ]
  }
}

That's it. Request, response. The AI gets the text back and incorporates it into its answer. The whole thing takes maybe 200ms.

The three things a server can expose

MCP servers aren't just about tools. There are three types of things a server can offer:

Tools are actions. "Fetch this URL." "Create a GitHub issue." "Run this SQL query." The AI decides when to call them. Each tool has a name, description, and a JSON Schema for its parameters.

Resources are data. Files, database records, API responses — things the AI can read. Each resource has a URI (like file:///path/to/doc.txt) and a MIME type. The AI can request specific resources by URI.

Prompts are templates. Pre-built prompt fragments that the server provides. A code review server might offer a "review this PR" prompt template that takes a PR URL as an argument. Less common than tools and resources, but useful for standardizing workflows.

Most servers you'll encounter only implement tools. But the protocol supports all three, and the handshake tells the IDE which ones are available.

How transport works

MCP currently uses stdio as its main transport. That means:

  • The IDE starts the server as a subprocess
  • Messages go in through the server's stdin
  • Responses come back through the server's stdout
  • Each message is one line of JSON, terminated by \n
  • The server's stderr is free for logging/debugging (doesn't interfere with the protocol)

This is surprisingly elegant. No ports to manage. No HTTP overhead. No authentication needed — it's all local. The process boundary is the security boundary. And because it's just stdin/stdout, it works with any language: Python, Node, Go, Rust, whatever can read a line and write a line.

There's also an SSE (Server-Sent Events) transport for remote servers, but most MCP servers today use stdio.

Try it yourself

You don't need an IDE to see MCP in action. Install mcptools and point it at any MCP server:

$ pip install git+https://github.com/jannik-cas/mcptools.git

# See what a server offers
$ mcptools inspect uvx mcp-server-fetch

┌───────────────────────── MCP Server ─────────────────────────┐
│ mcp-fetch v1.26.0                                    │
└──────────────────────────────────────────────────────────────┘
                           Tools (1)
  Name    Description                    Parameters
  fetch   Fetches a URL from the         url: string*,
          internet and extracts its      max_length: integer,
          contents as markdown.          raw: boolean

That command just did the full handshake I described above — initialize, notifications/initialized, tools/list — and formatted the result. Under the hood, it's the exact same JSON-RPC messages your IDE sends.

Or check if your existing config is healthy:

$ mcptools doctor

Config: ~/.claude/claude_desktop_config.json

  Checking fetch...          ✓ 1 tool, 1 prompt
  Checking github...         ✓ 35 tools
  Checking filesystem...     ✓ 5 tools, 2 resources

Summary: 3 healthy

This starts every server in your config, runs the handshake, counts capabilities, and measures latency. Two seconds and you know if everything works.

Where to find MCP servers

There are already hundreds of MCP servers out there. Some popular ones:

  • mcp-server-fetch — Fetch web pages and extract as markdown
  • mcp-server-github — Full GitHub API: repos, issues, PRs, code search
  • mcp-server-filesystem — Read/write files in a sandboxed directory
  • mcp-server-time — Current time and timezone conversions
  • mcp-server-sqlite — Query SQLite databases
  • mcp-server-postgres — Query PostgreSQL databases

Most are installable with uvx (Python) or npx (Node). The modelcontextprotocol GitHub org has the official ones. But anyone can build an MCP server — it's just a process that reads JSON from stdin and writes JSON to stdout.

The mental model

If you remember one thing from this post, let it be this:

Your IDE
    ↓ starts subprocess, sends JSON-RPC over stdin
MCP Server (just a process)
    ↓ responds over stdout
Your IDE
    ↓ gives the response to the AI model
AI decides what to do next

That's MCP. A subprocess that speaks JSON-RPC. The AI model sees the tool descriptions, decides when to use them, and the IDE handles the plumbing. The config file tells the IDE which servers to start. The protocol handles the rest.

Nothing more to it than that. The spec is surprisingly small for how much it enables.
