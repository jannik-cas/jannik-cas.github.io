MCP and Claude Code: What's Happening Under the Hood

March 11, 2026 · 9 min read

In my last post I explained what MCP is — the protocol that lets AI assistants call tools. Today I want to get specific about one tool I use daily: Claude Code, Anthropic's CLI agent. It leans heavily on MCP, and understanding that connection has changed how I work with it.

This isn't a Claude Code tutorial. It's about what happens between Claude Code and MCP servers — how the tool calls flow, where the config lives, and how to figure out what's going wrong when a tool isn't working.

Claude Code is an MCP client

When you run claude in your terminal, you're starting an MCP client. It reads your MCP config, launches server processes, does the JSON-RPC handshake I described in my last post, and then — here's the key part — it gives the model access to all those tools alongside its built-in ones.

So when Claude Code reads a file, runs a shell command, or searches your codebase, some of those actions go through MCP. The model sees a combined list of tools: built-in ones (like Read, Edit, Bash) plus whatever your MCP servers expose.

From the model's perspective, there's no difference. A tool is a tool. It has a name, a description, and parameters. Whether it's built into Claude Code or comes from an MCP server, the model calls it the same way.

Where Claude Code looks for MCP config

Claude Code checks several locations for MCP server configuration:

~/.claude/claude_desktop_config.json      # shared with Claude Desktop
~/Library/Application Support/Claude/claude_desktop_config.json  # macOS

The config format is the same one Claude Desktop uses. If you've already set up MCP servers for Claude Desktop, Claude Code picks them up automatically. No extra configuration.

Here's a real config I use daily:

{
  "mcpServers": {
    "fetch": {
      "command": "uvx",
      "args": ["mcp-server-fetch"]
    },
    "github": {
      "command": "uvx",
      "args": ["mcp-server-github"],
      "env": {
        "GITHUB_TOKEN": "ghp_your_token_here"
      }
    }
  }
}

Two servers. When Claude Code starts, it launches both as subprocesses, does the MCP handshake with each, and the model gets access to their tools.

What the tool call flow looks like

Say you're in Claude Code and you ask: "Look up the latest issues on the mcptools repo."

Here's what actually happens:

1. The model sees the GitHub MCP server's tools in its tool list. One of them is probably called list_issues or search_issues.

2. The model decides to call it and generates the arguments:

{
  "jsonrpc": "2.0",
  "id": 42,
  "method": "tools/call",
  "params": {
    "name": "list_issues",
    "arguments": {
      "owner": "jannik-cas",
      "repo": "mcptools",
      "state": "open"
    }
  }
}

3. The MCP server (mcp-server-github) receives this, calls the GitHub API with your token, and returns:

{
  "jsonrpc": "2.0",
  "id": 42,
  "result": {
    "content": [
      {
        "type": "text",
        "text": "[{\"number\": 1, \"title\": \"Add SSE transport support\", ...}]"
      }
    ]
  }
}

4. Claude Code feeds this back to the model, which formats a nice answer for you.

The whole round-trip is maybe 500ms. You just see the response in your terminal. The JSON-RPC, the subprocess, the GitHub API call — all invisible unless you go looking for it.

Adding an MCP server to Claude Code

Let's say you want Claude Code to be able to fetch web pages. Here's the setup, start to finish:

Step 1: Add the server to your config:

$ cat ~/.claude/claude_desktop_config.json
{
  "mcpServers": {
    "fetch": {
      "command": "uvx",
      "args": ["mcp-server-fetch"]
    }
  }
}

Step 2: Restart Claude Code. That's it. Next time you start claude, it'll launch the fetch server and the model will have a fetch tool available.

Step 3: Verify it works:

$ mcptools doctor --config ~/.claude/claude_desktop_config.json

Config: ~/.claude/claude_desktop_config.json

  Checking fetch...  ✓ 1 tool, 1 prompt

Green checkmark. The server starts, responds to MCP, and exposes one tool and one prompt. You're good.

Now in Claude Code you can say "fetch the content of https://example.com" and the model will use the MCP tool to do it.

When things break

This is the part that matters in practice. MCP tools in Claude Code fail silently. If a server doesn't start, or the token is wrong, or there's a timeout — you just don't see the tool. The model doesn't mention it. It's just... not there.

I've hit all of these:

  • Tool not showing up: The server command is wrong, or the binary isn't installed. mcptools doctor catches this instantly.
  • Tool shows up but fails: Usually a missing or expired token. The MCP server starts fine but chokes when it tries to call the external API. Use mcptools proxy --no-tui to watch the traffic and see the error response.
  • Tool is slow: Some servers (looking at you, GitHub with 35 tools) take 3+ seconds to initialize. mcptools doctor warns you about this.
  • Wrong arguments: The model sometimes generates arguments that don't match the schema. mcptools inspect shows you the exact schema so you can tell if the model is calling it correctly.

The debugging workflow I've settled on:

# 1. Is the server healthy?
$ mcptools doctor --config ~/.claude/claude_desktop_config.json

# 2. What tools does it actually expose?
$ mcptools inspect uvx mcp-server-github

# 3. Watch the live traffic
$ mcptools proxy --config ~/.claude/claude_desktop_config.json --server github --no-tui

Step 1 tells you if the server starts. Step 2 tells you what it offers. Step 3 shows you exactly what messages are flowing through. Between the three, I've never had an MCP issue I couldn't diagnose.

MCP servers I actually use with Claude Code

After a few months of using Claude Code with various MCP servers, here's what stuck:

mcp-server-fetch — The most useful one. Lets Claude grab documentation, check URLs, pull in context from the web. I use this daily.

mcp-server-github — Creates issues, reads PRs, searches code across repos. Slow to start (35 tools is a lot) but powerful once it's running. I mostly use it for "check the open issues on this repo" type tasks.

Custom project servers — I've written a couple of small MCP servers specific to my work. One queries our internal documentation. Another talks to a staging API. The beauty of MCP is that a server is just a script that reads stdin and writes stdout — you can build one in an afternoon.

Building a simple MCP server (5-minute version)

If you want Claude Code to do something specific, you can build an MCP server in about 50 lines of Python. Here's the skeleton:

from mcp.server import Server
from mcp.server.stdio import stdio_server

app = Server("my-server")

@app.list_tools()
async def list_tools():
    return [
        {
            "name": "greet",
            "description": "Say hello to someone",
            "inputSchema": {
                "type": "object",
                "properties": {
                    "name": {"type": "string"}
                },
                "required": ["name"]
            }
        }
    ]

@app.call_tool()
async def call_tool(name, arguments):
    if name == "greet":
        return [{"type": "text", "text": f"Hello, {arguments['name']}!"}]

async def main():
    async with stdio_server() as (read, write):
        await app.run(read, write)

import asyncio
asyncio.run(main())

Add it to your config:

{
  "mcpServers": {
    "my-server": {
      "command": "python",
      "args": ["my_server.py"]
    }
  }
}

Restart Claude Code, and you can say "greet Jannik" and the model will call your MCP server. Obviously this example is silly, but replace greet with "query our internal API" or "search our docs database" and it gets useful fast.

You can verify it works before even opening Claude Code:

$ mcptools inspect python my_server.py

┌───────────────────────── MCP Server ─────────────────────────┐
│ my-server                                             │
└──────────────────────────────────────────────────────────────┘
                           Tools (1)
  Name    Description              Parameters
  greet   Say hello to someone     name: string*

The bigger picture

What I find most interesting about MCP in Claude Code isn't the protocol itself — it's what it enables. Before MCP, if I wanted Claude Code to interact with something custom, I had to describe it in the prompt and hope for the best. Now I can give it actual tools that do real things, and it figures out when to use them.

The gap between "AI that can talk about your code" and "AI that can interact with your systems" is exactly what MCP fills. And because it's a standard protocol, any server you build works with any MCP client — not just Claude Code, but also Claude Desktop, Cursor, and anything else that supports MCP.

If you're using Claude Code and haven't set up any MCP servers yet, start with mcp-server-fetch. It takes two minutes and the difference is noticeable immediately.
