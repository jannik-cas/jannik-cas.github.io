I Built mitmproxy for MCP — Because Debugging MCP Servers Shouldn't Require Print Statements

March 9, 2026 · 8 min read · View on GitHub

Last month I lost 45 minutes to a typo. An environment variable in an MCP config. That's it. A typo. I couldn't tell because MCP gives you absolutely nothing when things go wrong — no errors, no logs, just... silence. So I built a tool to make sure that never happens again.

Everything below runs against real MCP servers. You can follow along if you want — takes about 5 minutes.

How I ended up here

I was wiring up three MCP servers for a side project. Two worked. The third one just... didn't. No error. No crash. Nothing in the IDE. Just a tool that should have been there and wasn't.

My debugging process, roughly:

1. Re-read the config JSON for the 12th time
2. Restart Claude Desktop
3. Wait 30 seconds
4. Still broken
5. Add print() to the server code
6. Realize prints go to stdout and break the JSON-RPC stream
7. Redirect to stderr
8. Restart again
9. Typo in the env var name
10. 45 minutes gone

A typo. GITHUB_TOKE instead of GITHUB_TOKEN. Forty-five minutes.

The thing that got me wasn't even the bug. It was that MCP gives you nothing to work with. No way to see what your IDE is sending. No way to check if the server even starts. You're just staring at a JSON config file hoping you'll spot the problem before your coffee goes cold.

So I built the thing I kept wishing existed.

Install

$ pip install git+https://github.com/jannik-cas/mcptools.git

That's the whole setup. Let me show you what it does.

mcptools doctor — find out what's broken

Here's a config with three servers. Two work, one is intentionally broken. Save it as test_config.json or grab the one from the repo:

{
  "mcpServers": {
    "fetch": {
      "command": "uvx",
      "args": ["mcp-server-fetch"]
    },
    "time": {
      "command": "uvx",
      "args": ["mcp-server-time"]
    },
    "broken": {
      "command": "nonexistent-mcp-server-xyz",
      "args": []
    }
  }
}

Run it:

$ mcptools doctor --config examples/demo_config.json

Config: examples/demo_config.json

  Checking fetch...          ✓ 1 tool, 1 prompt
  Checking time...           ✓ 2 tools
  Checking broken-server...  ✗ Command not found: nonexistent-mcp-server-xyz
    → Ensure 'nonexistent-mcp-server-xyz' is installed and in your PATH

Summary: 2 healthy, 1 error

Two seconds. It found the broken server, told me exactly what's wrong, and even told me how to fix it.

Under the hood, doctor actually starts each server process, does the full MCP handshake, queries capabilities, and measures how long it all takes. So it catches the stuff that's hard to spot manually — typos in commands, unset env vars, servers that start but don't respond to MCP, slow servers, init failures.

Try it on your own setup. If you use Claude Desktop: mcptools doctor --config ~/.claude/claude_desktop_config.json. Cursor: mcptools doctor --config ~/.cursor/mcp.json. Or just run mcptools doctor without arguments — it'll look in the usual places for Claude Desktop, Cursor, VS Code, and Windsurf.

mcptools inspect — see what a server offers

Say you found an MCP server on GitHub and want to know what it actually does before wiring it into your IDE. What tools? What parameters?

$ mcptools inspect uvx mcp-server-fetch

┌───────────────────────── MCP Server ─────────────────────────┐
│ mcp-fetch v1.26.0                                    │
└──────────────────────────────────────────────────────────────┘
                           Tools (1)
  Name    Description                    Parameters
  fetch   Fetches a URL from the         url: string*,
          internet and extracts its      max_length: integer,
          contents as markdown.          start_index: integer,
                                         raw: boolean
                          Prompts (1)
  Name    Description                    Arguments
  fetch   Fetch a URL and extract         url*
          its contents

One tool, one prompt, all the parameter types laid out. The * means required. No more guessing from README examples that may or may not be up to date.

Works with anything that speaks MCP over stdio — Python, Node, Go, whatever:

$ mcptools inspect uvx mcp-server-time
$ mcptools inspect python my_server.py
$ mcptools inspect npx @modelcontextprotocol/server-filesystem /tmp
$ mcptools inspect node build/index.js

mcptools proxy — watch the traffic

This is the one I use the most. mcptools proxy sits between your IDE and the MCP server, like a wireshark for MCP. Every JSON-RPC message, both directions, with timing.

Your IDE (Claude Code / Cursor / Windsurf)
    ↓ stdin/stdout
mcptools proxy  ←  sees everything
    ↓ stdin/stdout
MCP Server

$ mcptools proxy --config ~/.claude/claude_desktop_config.json --server fetch --no-tui

>>> initialize                        120ms
<<< initialize
>>> notifications/initialized
>>> tools/list                        45ms
<<< tools/list
>>> tools/call                        1205ms
<<< tools/call

That's the --no-tui log mode. By default it opens a full TUI dashboard with a live message table, latency per call, error highlighting, stats, and a detail panel where you can click any row to see the raw JSON-RPC payload. Honestly, just seeing the latency numbers next to each call has been worth the whole project for me.

mcptools record + replay — capture sessions

Sometimes you need to show someone else what happened. Or reproduce something that only happens when you're not looking at the logs. Record captures everything to JSON, replay plays it back.

$ mcptools record --config ~/.claude/claude_desktop_config.json -o debug_session.json

Recording: fetch
Output: debug_session.json
Ctrl+C to stop and save

$ mcptools replay debug_session.json --speed 2 --filter "tools/*"

The JSON file has timestamps, directions, full payloads, latency — everything you'd want in a bug report. Way better than "it didn't work" with a screenshot of your IDE.

Putting it all together: a real debugging session

Here's how I actually used these tools last week when a GitHub MCP server kept timing out.

Step 1: Is it even connecting?

$ mcptools doctor --server github

  Checking github...  ⚠ 35 tools — Slow response (3200ms)

It connects, but 3.2 seconds for an init? That's suspicious.

Step 2: What's in there?

$ mcptools inspect uvx mcp-server-github

┌───────────────────────── MCP Server ─────────────────────────┐
│ github-mcp-server                                  │
└──────────────────────────────────────────────────────────────┘
                         Tools (35)
  ...

35 tools. OK, that explains the slow init — probably doing a lot of work at startup. But the real issue was the timeout during use.

Step 3: Watch what's actually happening

$ mcptools proxy --server github --no-tui

>>> initialize                           3205ms
<<< initialize
>>> tools/list                           89ms
<<< tools/list
>>> tools/call                           timeout
  ERROR: Request timed out after 30s

There it is. The tools/call times out after 30 seconds. Now I know it's not a config issue — the server starts fine, lists tools fine, but chokes on the actual call. Probably a token scope issue or a rate limit. Either way, I know exactly where to look now.

Step 4: Capture it for the bug report

$ mcptools record --server github -o github_timeout.json
# reproduce the issue, then Ctrl+C

Session saved: github_timeout.json
8 messages recorded

Attach github_timeout.json to the issue. The maintainer gets the full trace — what was sent, what came back, timing, everything. Way more useful than "it doesn't work."

How it works

The core idea is simple: MCP uses JSON-RPC 2.0 over stdio. So mcptools just speaks the same protocol.

inspect starts the server as a subprocess, runs the MCP handshake, asks for tools/resources/prompts, and formats what comes back. proxy reads from stdin (your IDE), forwards to the server, reads responses, forwards back. Your IDE has no idea it's there — it's just a pipe in the middle that happens to log everything passing through.

src/mcptools/
├── cli.py              # Click CLI — 5 commands
├── jsonrpc.py          # Shared JSON-RPC helpers
├── config/parser.py    # Auto-detects IDE configs
├── doctor/checks.py    # Concurrent health checks
├── inspect/server.py   # Server introspection
├── proxy/
│   ├── interceptor.py  # Traffic interception
│   └── transport.py    # Stdio transport layer
├── record/
│   ├── recorder.py     # Session capture
│   └── replayer.py     # Session replay
└── tui/dashboard.py    # Real-time TUI

The whole thing is about 2,500 lines of Python. Nothing fancy. Textual for the TUI, Rich for the pretty tables, Click for the CLI, Pydantic for message validation.
